## Project architecture

Neutrino follows a **clean architecture** pattern with clear separation of concerns between backend and frontend:

```
neutrino/
├── backend/           # Python API + Streamlit UI
├── frontend/          # React landing + dashboard
├── docs/              # Documentation (Mintlify)
└── .env               # Environment variables (gitignored)
```

## Backend structure

The backend is organized as a Python package with **51 Python files** across multiple modules:

```
backend/
├── src/advisor/                # Main application package
│   ├── config/                 # Configuration and settings
│   ├── database/               # Database layer (Supabase)
│   ├── github/                 # GitHub API integration
│   ├── llm/                    # LLM client (OpenRouter)
│   ├── analysis/               # Intelligence engine
│   ├── trends/                 # Trend intelligence (RAG)
│   ├── reports/                # Report generation
│   ├── api/                    # FastAPI endpoints
│   └── ui/                     # Streamlit UI
├── tests/                      # Test suite (60 tests)
├── migrations/                 # Database migrations
├── scripts/                    # Utility scripts
├── pyproject.toml              # Dependencies and config
└── uv.lock                     # Dependency lock file
```

## Config module

**Location:** `backend/src/advisor/config/`

Manages application configuration with environment variable validation.

<Accordion title="settings.py">
  Pydantic-based settings with environment loading:

  ```python
  from pydantic_settings import BaseSettings

  class Settings(BaseSettings):
      # Supabase
      supabase_url: str
      supabase_service_role_key: str
      
      # OpenRouter (multi-key rotation)
      openrouter_api_key_1: str
      openrouter_api_key_2: str | None = None
      openrouter_api_key_3: str | None = None
      openrouter_api_key_4: str | None = None
      
      # Serper (web search)
      serper_api_key: str | None = None
      
      @property
      def openrouter_api_keys(self) -> list[str]:
          """Get all available API keys for rotation."""
          return [k for k in [self.openrouter_api_key_1, ...] if k]
  ```

  **Features:**
  - Automatic `.env` file loading
  - Fail-fast validation for required variables
  - Multi-key rotation support
  - Type-safe configuration access
</Accordion>

**Key file:** `config/settings.py:1`

## LLM module

**Location:** `backend/src/advisor/llm/`

Manages LLM interactions via OpenRouter with optimization features.

<AccordionGroup>
  <Accordion title="client.py">
    OpenRouter client with advanced features:

    ```python
    class OpenRouterClient:
        def __init__(self, api_keys: list[str]):
            """Initialize with shared httpx.AsyncClient."""
            self._client = None  # Lazy-initialized
            self._api_keys = api_keys
            self._current_key_index = 0
        
        async def complete(
            self,
            prompt: str,
            system_prompt: str,
            temperature: float = 0.7
        ) -> dict:
            """Generate completion with timing."""
            # Returns: {content, usage, duration_ms, model}
        
        async def close(self):
            """Close shared HTTP client."""
    ```

    **Optimizations:**
    - Shared `httpx.AsyncClient` (eliminates TCP/TLS overhead)
    - Multi-key rotation (up to 4 keys)
    - Model fallback (primary → secondary)
    - Per-call `duration_ms` tracking
    - Token usage monitoring
  </Accordion>

  <Accordion title="models.py">

## Key optimizations

1. **Shared httpx Client** - Single `AsyncClient` reused across all LLM calls
2. **Parallel everything** - RAG lookups, trend searches, file fetches via `asyncio.gather`
3. **RAG-first flow** - Check pgvector cache before collecting fresh data
4. **Strategic fetching** - Smart 3-pass file retrieval (~150 files)
5. **Multi-key rotation** - Up to 4 OpenRouter API keys
6. **Per-call timing** - Every LLM/RAG call records `duration_ms`
