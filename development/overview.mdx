## Getting started

Neutrino is built with a clean architecture using Python (backend) and React (frontend). Follow this guide to set up your development environment and understand the contribution workflow.

### Prerequisites

Before you begin, ensure you have:

- **Python 3.11+** for backend development
- **Node.js 18+** and npm for frontend development
- **Git** for version control
- **uv** package manager (recommended for Python dependencies)

### Environment setup

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/your-org/neutrino.git
    cd neutrino
    ```
  </Step>

  <Step title="Set up backend environment">
    Navigate to the backend directory and install dependencies:

    ```bash
    cd backend
    uv sync
    ```

    Create a `.env` file with required environment variables:

    ```env
    # Required
    SUPABASE_URL=https://your-project.supabase.co
    SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

    # Required (at least one)
    OPENROUTER_API_KEY_1=sk-or-v1-xxx
    OPENROUTER_API_KEY_2=sk-or-v1-xxx  # Optional
    OPENROUTER_API_KEY_3=sk-or-v1-xxx  # Optional
    OPENROUTER_API_KEY_4=sk-or-v1-xxx  # Optional

    # Required for trend data collection
    SERPER_API_KEY=your-serper-api-key
    ```
  </Step>

  <Step title="Set up frontend environment">
    Navigate to the frontend directory and install dependencies:

    ```bash
    cd ../frontend
    npm install
    ```
  </Step>

  <Step title="Start development servers">
    Start the backend API server:

    ```bash
    cd backend
    uv run uvicorn advisor.api.endpoints:app --reload --port 8000
    ```

    In a separate terminal, start the frontend dev server:

    ```bash
    cd frontend
    npm run dev
    ```

    Access the application:
    - **Frontend:** http://localhost:5173
    - **API:** http://localhost:8000
    - **API Docs:** http://localhost:8000/docs
  </Step>
</Steps>

## Contribution workflow

### Branch naming conventions

Follow these naming conventions for branches:

- **Feature branches:** `feature/description-of-feature`
- **Bug fixes:** `fix/description-of-bug`
- **Documentation:** `docs/description-of-change`
- **Refactoring:** `refactor/description-of-change`
- **Tests:** `test/description-of-test`

<CodeGroup>
```bash Feature branch
git checkout -b feature/add-pdf-export
```

```bash Bug fix branch
git checkout -b fix/github-auth-error
```

```bash Documentation branch
git checkout -b docs/update-api-guide
```
</CodeGroup>

### Git workflow

<Steps>
  <Step title="Create a branch">
    Always create a new branch from `main` for your changes:

    ```bash
    git checkout main
    git pull origin main
    git checkout -b feature/your-feature-name
    ```
  </Step>

  <Step title="Make your changes">
    Write code following our style guidelines and ensure all tests pass.
  </Step>

  <Step title="Run linting and formatting">
    Before committing, run linting and formatting checks:

    ```bash
    uv run ruff check src/ && uv run ruff format src/
    ```
  </Step>

  <Step title="Run tests">
    Ensure all tests pass before submitting:

    ```bash
    uv run pytest tests/ -v
    ```

    See [Testing](/development/testing) for more details.
  </Step>

  <Step title="Commit your changes">
    Write clear, descriptive commit messages:

    ```bash
    git add .
    git commit -m "feat: add PDF export functionality"
    ```

    Use conventional commit prefixes:
    - `feat:` for new features
    - `fix:` for bug fixes
    - `docs:` for documentation
    - `refactor:` for code refactoring
    - `test:` for test changes
    - `chore:` for maintenance tasks
  </Step>

  <Step title="Push and create a pull request">
    Push your branch and create a pull request:

    ```bash
    git push origin feature/your-feature-name
    ```

    Open a pull request on GitHub with:
    - Clear description of changes
    - Reference to any related issues
    - Screenshots for UI changes
  </Step>
</Steps>

## Code quality standards

### Linting and formatting

We use **Ruff** for Python linting and formatting. Run these commands before committing:

```bash
# Check for linting issues
uv run ruff check src/

# Auto-format code
uv run ruff format src/

# Run both checks
uv run ruff check src/ && uv run ruff format src/
```

<Accordion title="Ruff configuration">
  Our Ruff configuration in `pyproject.toml`:

  ```toml
  [tool.ruff]
  line-length = 88
  target-version = "py311"

  [tool.ruff.lint]
  select = ["E", "F", "I", "W", "UP", "B", "C4", "SIM"]
  ignore = ["E501"]

  [tool.ruff.lint.isort]
  known-first-party = ["advisor"]
  ```
</Accordion>

### Frontend code quality

For frontend development:

```bash
# Run ESLint
npm run lint

# Build for production (checks for errors)
npm run build
```

## Testing requirements

All contributions must include appropriate tests:

- **New features** must include unit tests and integration tests where applicable
- **Bug fixes** should include tests that verify the fix
- **Code coverage** should not decrease with your changes

Run the test suite:

```bash
# Run all tests
uv run pytest tests/ -v

# Run with coverage report
uv run pytest tests/ --cov=advisor --cov-report=html
```

See the [Testing guide](/development/testing) for detailed information about our test suite.

### Documentation

When adding new features:

- Update relevant documentation in the `/docs` directory
- Add docstrings to all public functions and classes
- Include code examples where appropriate
- Update the README if adding user-facing features

### Environment variables

Never commit sensitive information:

- Use `.env` files for local development (already gitignored)
- Document required environment variables
- Use Pydantic settings for validation
- Keep credentials ephemeral (never stored in database)

<Warning>
  The `.env` file should never be committed to version control. Always use `.env.example` as a template.
</Warning>
